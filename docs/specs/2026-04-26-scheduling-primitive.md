# CNS v0.2 Scheduling Primitive — Design Spec

**Status:** Draft (design only; implementation deferred until first one-shot RemoteTrigger fires successfully on 2026-04-30)
**Author:** James Gao + Claude
**Date:** 2026-04-26
**Tracks:** Brain/Bets/bet_cns_scheduling_primitive.md (in vault — install repo)

## Problem

CNS has gained a scheduling pattern but no primitive that captures it. Today the user's vault contains three hand-coded scheduled jobs — each is a sibling directory under `automation/` with a `prompt.md`, a `trigger.json` (RemoteTrigger create body), and a `README.md`:

- `automation/daily-ceo-report/` — recurring synthesis (every night at 11pm PT)
- `automation/2026-04-30-decision-day/` — one-shot decision-day brief
- `automation/2026-08-18-dylan-prereqs/` — one-shot prerequisites assessment

These three jobs share ~80% of their structure: launchd-equivalent scheduling on Anthropic's `RemoteTrigger` API; a stub-first defensive prompt pattern; push verification with GitHub-issue fallback; for one-shots, a self-disable step. They diverge on shape (recurring vs one-shot, synthesis vs action), context (which repos/dirs to clone), and prompt content.

Without a primitive, every new scheduled prompt requires hand-coding the full RemoteTrigger create body + prompt scaffolding, and bet `kill_criteria` with embedded date references ("April 30 go/no-go", "May 1 notice", "August prerequisites check") never become machine-extractable scheduling intent.

## Goals

1. **Express both existing job shapes (recurring synthesis + one-shot action) as instances of one config-driven abstraction.**
2. **Hide the RemoteTrigger create-body assembly** behind a CLI surface that takes a small declarative config and generates the API call.
3. **Allow CNS state to be injected into prompts** — `{{cns_active_bets}}`, `{{cns_bet:slug}}`, `{{cns_open_conflicts}}`, `{{cns_upcoming_deadlines}}` — so scheduled prompts can reference the bet system without each prompt fetching state manually.
4. **Make one-shot semantics first-class** so users don't have to hand-roll the self-disable curl call.

## Non-goals (v0.2)

- **Conditional/event-driven triggers** ("fire when bet X's last_reviewed > 30 days"). Time-based scheduling only in v0.2; conditional is a v0.3 problem.
- **Backends other than RemoteTrigger.** v0.2 ships against Anthropic's RemoteTrigger API only. If `RemoteTrigger` becomes unavailable or another scheduler emerges, the abstraction is shaped to allow a backend swap, but no second backend is built in v0.2.
- **LLM-judge mode in detector** (out of scope for this spec).
- **Auto-scheduling from bet kill_criteria date references.** The primitive supports manual `cns schedule add`; auto-extracting dates from bets is a v0.3 nice-to-have once the primitive shape is validated.

## Design

### Directory layout

Each scheduled job lives in `<vault>/automation/<job-name>/`:

```
automation/<job-name>/
├── config.yaml      # the declarative scheduling spec (NEW — primary surface)
├── prompt.md        # the prompt template, with optional {{...}} injection markers
├── README.md        # human-readable explainer (optional, copy from template)
└── trigger.json     # RemoteTrigger create body (GENERATED by `cns schedule sync`)
```

The hand-edited surfaces are `config.yaml` and `prompt.md`. `trigger.json` is generated and should not be hand-edited (a comment in the file says so).

### `config.yaml` schema

```yaml
# Required
name: "Daily CEO Report — Gigaflow"          # human label; appears in claude.ai trigger UI
schedule:
  type: recurring                             # "recurring" or "one-shot"
  cron: "0 6 * * *"                           # standard 5-field cron, UTC
  # For one-shot: primarily encodes via specific date in cron. Add:
  # auto_disable_after_fire: true            # default true for one-shot

prompt: prompt.md                             # path to prompt template (relative to job dir)

# Required for RemoteTrigger backend; could be different for other backends.
backend:
  kind: remote_trigger
  environment_id: env_018akjJuQRFoDDHDTWCgMn4d
  model: claude-opus-4-7
  persist_session: false
  allowed_tools: [Bash, Read, Write, Edit, Glob, Grep]
  sources:
    - git_repository: https://github.com/Jamesjg2/gigaflow-planning
    - git_repository: https://github.com/GigaFlow-AI/gigaflow      # optional, multiple OK

# Optional: which CNS-aware values to substitute into prompt.md
inject:
  - cns_active_bets                           # full BETS.md as markdown
  - cns_open_conflicts                        # CONFLICTS.md
  - cns_upcoming_deadlines                    # parsed from kill_criteria
  - cns_bet:bet_neurips_db_sprint             # specific bet (one or more)
  - vault_recent_commits                      # git log over the last detection window
```

Field-by-field:

- `name` — required. Used as the trigger's `name` in claude.ai's UI.
- `schedule.type` — `recurring` (default) | `one-shot`. Affects only the auto-disable behavior post-fire; both compile to a cron expression on RemoteTrigger.
- `schedule.cron` — standard 5-field cron in UTC. Required.
- `schedule.auto_disable_after_fire` — bool. Default: `true` for one-shot, `false` for recurring. When `true`, the generated prompt has a final step that calls `RemoteTrigger update enabled: false` after a successful push. (Same self-disable pattern we hand-rolled for the April 30 + Aug 18 triggers.)
- `prompt` — relative path to a markdown template. Required.
- `backend.kind` — `remote_trigger` only in v0.2.
- `backend.environment_id` / `model` / `persist_session` / `allowed_tools` / `sources` — pass-through to the RemoteTrigger create body.
- `inject` — list of CNS injection markers. The CLI generates a "context block" markdown header that prepends to `prompt.md` in the generated `trigger.json` (see "Context injection" below).

### CLI surface

```
cns schedule init <job-name>           # scaffold a new job dir with config.yaml + prompt.md + README.md from templates
cns schedule list                      # list all jobs in automation/, with their RemoteTrigger IDs and next_run_at
cns schedule sync [<job-name>]         # render trigger.json from config.yaml + prompt.md + injection; create-or-update RemoteTrigger
cns schedule status [<job-name>]       # query RemoteTrigger API for current state (enabled, last fire, next_run_at)
cns schedule disable <job-name>        # set enabled: false on the live trigger
cns schedule delete <job-name>         # delete the live trigger AND optionally remove the job dir
cns schedule run <job-name>            # manually fire the trigger NOW (passthrough to RemoteTrigger run action)
```

The single load-bearing verb is `sync` — it's idempotent: rendering the same `config.yaml` twice produces the same `trigger.json` and either creates or updates the RemoteTrigger. Users run `sync` after editing `config.yaml` or `prompt.md`.

### Context injection

When `sync` renders `trigger.json`, it walks the `inject:` list and prepends a markdown context block to the prompt content sent to RemoteTrigger:

```markdown
<!-- BEGIN CNS CONTEXT INJECTION (auto-generated, do not edit) -->

## CNS Active Bets

[contents of BETS.md, abbreviated to first ~40 lines]

## CNS Open Conflicts

[contents of CONFLICTS.md]

## CNS Bet: bet_neurips_db_sprint

[full bet file contents]

## Vault Recent Commits (last 24h)

[git log output]

<!-- END CNS CONTEXT INJECTION -->

[user's prompt.md content here]
```

This means the cloud agent receives all CNS state inline at fire time without having to fetch it itself. The injection is regenerated each `sync` from current vault state — the trigger is always synced to the *latest* CNS context, but only updates when the user runs `sync`.

For one-shot triggers that need fresh data at fire time (e.g., the Aug 18 trigger should read bet state *as of Aug 18*, not as of Apr 26), the `inject:` field should be empty and the prompt should fetch state itself (which is what the Aug 18 prompt does today). For recurring synthesis (daily-ceo-report), injection at sync time + lazy refresh in the agent is appropriate.

**Trade-off:** injection couples the trigger to vault state at sync time. That's the right default for recurring jobs (they sync nightly via the auto-loop too) and the wrong default for one-shots that fire weeks later. Hence `inject:` is opt-in, empty by default.

### Backend abstraction

`backend.kind: remote_trigger` is the only supported backend in v0.2, but the schema is shaped so a future `backend.kind: launchd` (local macOS scheduling) or `backend.kind: github_actions` (CI cron) could plug in without breaking the config-yaml shape. Each backend would implement `render_trigger(config, prompt) -> opaque_payload` and `sync(payload) -> trigger_id`.

For v0.2, no backend abstraction layer is built — we ship the RemoteTrigger flow directly. The non-goal is "abstract over backends prematurely"; we just leave room for it.

### Migration plan

After v0.2 ships, the existing three jobs migrate from hand-coded to config-driven:

1. **`automation/daily-ceo-report/`** — write `config.yaml` matching the live trigger (`trig_01KW74dv6UqinH17UL7uYXKR`), keep `prompt.md` as-is, run `cns schedule sync daily-ceo-report`. The generated `trigger.json` should be byte- or behaviorally-identical to the existing hand-coded one.
2. **`automation/2026-04-30-decision-day/`** — same migration. The `auto_disable_after_fire: true` config makes the hand-rolled self-disable step in `prompt.md` redundant; it can be deleted post-migration.
3. **`automation/2026-08-18-dylan-prereqs/`** — same.

Migration test: for each job, `sync` should produce a `trigger.json` whose `events[0].data.message.content` matches the live trigger's stored content (modulo whitespace and the now-removed self-disable step).

### Risk: schema lock-in

Once users have config files in their vaults, schema changes become migrations. Mitigations:

- Version the schema explicitly: `cns_schedule_version: "0.2"` at top of every config. Future versions can detect and warn/migrate.
- Be conservative on what's *required* in v0.2. Only `name`, `schedule.cron`, `prompt`, `backend.kind`, and required RemoteTrigger fields are required. Everything else (auto_disable_after_fire, inject, etc.) is optional with defensible defaults.

### Risk: RemoteTrigger evolving under us

Anthropic could add native `one_shot: true` or `context_injection: [...]` fields, making part of this abstraction redundant. If/when that happens:

- The CNS abstraction stays useful as the *vault-side* representation (config files in version control, alongside bets) — even if it becomes a thinner shim over the API.
- We don't pre-emptively code against hypothetical RemoteTrigger features. v0.2 maps to today's RemoteTrigger; v0.3 reassesses.

## What's deliberately NOT in v0.2

- Auto-extraction of date references from bet `kill_criteria` to suggest schedules. Defer to v0.3 once we see how `cns schedule` is used in practice.
- A daemon or watchdog that re-syncs configs automatically. Users run `sync` manually after edits.
- Conditional triggers ("fire when X happens"). Time-based only.
- Backends other than RemoteTrigger.
- LLM-rendered prompts (every prompt is a static markdown template plus context injection; no LLM-in-the-loop at sync time).

## Validation plan

After implementation:

1. **Migration test:** all three existing jobs migrate cleanly, generated `trigger.json` matches live state.
2. **End-to-end test:** scaffold a new test job (`cns schedule init test-job`), edit config + prompt, `sync`, manually fire via `cns schedule run`, verify the agent fires correctly, then `delete`.
3. **Schema-roundtrip test:** load a `config.yaml`, render `trigger.json`, parse trigger.json back to confirm semantic equivalence.
4. **Idempotency test:** running `sync` twice in a row with no changes should produce the same `trigger.json` and not modify the live trigger.

## Decision log

- **Why config.yaml not config.json?** YAML matches CNS's existing config style (`.cns/config.yaml`). Lower friction.
- **Why a per-job directory rather than a single `automation.yaml`?** Per-job dirs keep prompt.md + README + trigger.json next to the config that describes them. Single-file would force separating `prompts/` from `automation.yaml`, which the existing hand-coded pattern already showed doesn't scale.
- **Why is `sync` user-invoked rather than a watchdog?** The user's vault uses Obsidian Git auto-push; there's no daemon to attach a file-watcher to. Manual `sync` is unambiguous and matches the existing `cns reindex` cadence (run after edits).
- **Why generate `trigger.json` rather than write directly to RemoteTrigger from `config.yaml`?** Two reasons: (1) the user can review the rendered trigger.json before sync hits the API, and (2) the trigger.json provides a reproducible artifact for "recreate this trigger from scratch" scenarios — same role today's hand-coded `trigger.json` files play.

## Open questions for review

1. **Should `sync` write the rendered context-injection back into prompt.md or only into the generated trigger.json?** Default: only into trigger.json (keep prompt.md as a clean template). Surface the rendered prompt via `cns schedule show <job>` if the user wants to inspect.
2. **What's the right default for `inject` if the user doesn't specify?** Probably empty. Don't auto-inject everything — too much context noise. Force users to opt-in per-injection.
3. **Should one-shots be auto-cleaned (delete the trigger record) after fire, or just disabled?** Lean: disable, leave the record as a tombstone. Users can `cns schedule delete` manually if they want full removal. Disabled triggers are visible in claude.ai's UI as a record of "this fired and ran".
4. **How do we handle the case where the user edits the live trigger via claude.ai's UI without running `sync`?** Detect drift on next `sync`, warn, offer to overwrite or reconcile. Defer detailed UX to implementation.
